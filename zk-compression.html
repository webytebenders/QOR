The Core Problem

The Panopticon Problem

Compliance data on a public ledger creates a financial surveillance machine. QOR eliminates this by design.

Surveillance Nightmare

Storing ISO 20022 XML (9,000+ characters) on a public ledger exposes every remittance detail to anyone with a block explorer. A financial panopticon.

Regulatory Violation

Public on-chain storage of personal financial data violates GDPR, financial privacy laws, and data protection regulations across jurisdictions.

Astronomical Cost

Solana account rent for raw XML storage at 9,000+ chars per transaction makes on-chain compliance data economically impossible at scale.

The Real Need

Compliance WITHOUT surveillance. Institutions need proof of regulatory adherence, but users deserve privacy. These must coexist.

Public Ledger

All-Seeing Eye

The Solution

How ZK Compression Works

Three steps. Full compliance. Zero exposure. From 9,000+ characters of sensitive XML to a 128-byte on-chain footprint.

Step 01

Hash

The protocol takes a cryptographic fingerprint of sensitive remittance data. The full ISO 20022 XML is reduced to a deterministic, irreversible hash digest.

Cryptographic Fingerprint
Step 02

Compress

Only a 128-byte Merkle state root is stored on-chain via the Solana Metadata Pointer extension. The entire compliance payload compressed into a single verifiable root.

Merkle State Root
Step 03

Store

Actual plaintext XML is stored off-chain in a permissioned Data Availability (DA) layer. Only authorized counterparties with proper credentials can access the full data.

Permissioned DA Layer

The Photon Indexer

Open-source infrastructure for succinct ZK proofs of compliance — without revealing the underlying data.

Prove Everything. Reveal Nothing.

QOR leverages the open-source Photon indexer for state transitions, enabling users to generate succinct zero-knowledge proofs of compliance that can be verified on-chain in milliseconds — without exposing any raw personal data.
